Welcome to the Mark's Code Fractal and MSS (Mark Stephen Sobkow) Code Factory project pages. Mark's Code Fractal is the successor to MSS Code Factory, a major refresh of the technology and approach used for the code base which really shares very little with the prior knowledge base and code structure that produced the 2.13 code architecture.

MSS Code Factory 2.13 alpha07 is now available at https://github.com/msobkow/org.msscf.msscf.v2_13-installer/releases/tag/alpha07. You can find installation instructions at https://github.com/msobkow/org.msscf.msscf.v2_13-installer/ and the README.md displayed there. There have also been releases of the 2.13 knowledge base and model set as of 2026-06-30 10:00 CST, tagged as snapshot-202606301000 which are included in the installer and are required alongside the executable jars to run the tool. Alpha07 begins using the CodeVis attributes in the CFSec 3.1 model to produce a cfsecpub and cfsecpubobj object interface specifications. The code produced is incomplete and does not compile, but this whole process requires periodic snapshots so that the engine in "production" is capable of running the most recent knowledge bases.

Enhancements in Alpha07 Code for 3.1 2026-03-30 10:00 CST

The CFSec model has been liberally decorated with CodeVis="Public|Protected|Private" attributes as seems reasonable for distinguishing between objects, data, and relationships that should be available to inheriting/referencing applications and schemas (Public), available to application servers/services and user interfaces (Protected), or reserved for internal access by implementatiion code on the back-end server to ensure data integrity is maintained during all processing. Currently the first drafts of cfsecpub, cfsecpubobj, cfintpub, cfintpubobj, cfbampub, and cfbampubobj have been produced, but more work is required to properly hide non-public information in the PubObj object interfaces to be implemented by the access layer.

Once I'm content that the Pub and PubObj interface specifications are "complete" and "clean enough", I'll clone and modify them to produce the Protected interfaces as cfsecprot, cfsecprotobj, cfintprot, cfintprotobj, cfbamprot, and cfbamprotobj. The *prot packages already exist, but only have the authorization object inheritance hierarchy coded so far. This results in the build failures for the current version of cfsec*, because I need all the layers to be in place before I can flesh out the implementations of the interfaces by applying the same "IsPublic" and "IsProtected" verb based filtering of data for things like CFSecPubBuffSchema that implements the ICFSecPubSchema interface, and CFPubBuff[Table]Factory that implements ICFSecPub[Table]Factory.

So there's a long time before there is going to be a clean build again, and I haven't forgotten about the Role implementations; this current task just sounded like more fun to work on, and entertainment value is big on my priority list as I'm working on this while "retired" from life. Not that I've given up on ever "striking it rich" through my own efforts; I just realistically assess my odds of such rewards as less than the chances of winning Lotto 6/49 in Canada.

MAJOR CHANGE FOR 3.1 2026-06-17 13:40 CST

Moved the *buff packaging into the base interface packages. Buff instances are always going to be required, because I've decided to make all of the APIs talk Buff instances. The JPA layer will provide Buff interface methods as it does now, but also alternately-named JPA interface methods that its code can use directly with JPA instances during the processing of a transaction/service method. When returning in the JpaTable classes that do the security checks, those returned JPA instances will get converted to Buff instances for the upper application layer. The server method bodies will assume Buff access methods in its code, as everything except the JPA layer will be using buffers.

Major Enhancements In 2.13 alpha02 and 3.1

Mark's Code Fractal 3.1 was to introduce the concept of "tweaks", in the form of "schematweak [TweakName]", "indextweak [TweakName]" and "tabletweak [TweakName]" GEL verbs that lookup tweak specifications in the CFBam model dynamically rather than relying on hard-coded expansion verbs with unique names bound to the engine at compile time as the 2.13 code base had been doing. All those custom verbs are going bye-bye in 3.1.

The idea of "tweaking" code was so convenient that it got back-ported to 2.13, and is available now in parallel with the old approach of hard-coded verbs.

A "SchemaRole" specification within a Bam's SchemaDef now provides the means of defining roles that enable a list of security groups and roles, and which include/are auto-enabled by another set of security groups. Roles themselves are implemented as system security group objects in the security schema; all security group types and role types must have a unique name in the system security group set of definitions in the cluster. Currently the implementation of the roles is lacking; only the specification of them has been coded so far. Implementing them is going to take a bit of doing, and then I've got to flesh out the forget code for the security schema base cache and have the appropriate pieces of the JPA code base trigger forgetting of the various groups and roles that they've been modifying in or removing from persistent storage.

Major Enhancements Planned for 3.1

I will be adding a "CodeVisibility" specification to a Table specification for 3.1 to be used in the knowledge base that produces the 3.2 code. It will have a default value of "public" and could optionally be set to "protected". Public tables would be used as they are now, but protected tables would only be visible within a schema's implementation code, not accessed by any SchemaDef's that import the schema through a SchemaRef.

It may be necessary to extend the idea of code visibility to other aspects of the code base, such as the schema methods and the indexes and methods of tables, but even hiding the use of tables would require substantial modification of the knowledge base. There are a number of security tables that I'd really like to have protected from access by referencing schema definitions, which is what inspired this capability enhancement plan.

The concept of public and protected tables would also extend to the specification of public and protected schema specifications, with the protected schema interface extending the public schema interface, and the single implementations of the layers for the schema providing both interfaces. Then the referencing schema definitions and rules would only reference the public interface, and never access anything through the protected interface as a "short cut" or "performance tweak." There would also be public and protected schema table interfaces, with the protected methods and protected index accessors in the protected table interface, and all the others in the existing public interface.

The complexity of splitting the schema interfaces and referencing access in such a fashion is why I decided to put off this feature as a 3.1 implementation towards producing a 3.2 code base rather than considering adding it to my current 3.1 schedule. 3.1.42 will not be "Life, The Universe, and Everything" as I had planned, after all.

I'll add the specifications to the 3.1 BAM model for the feature soon, however.

Overview Of The Technology

Mark's Code Fractal (formerly MSS Code Factory) is a tool used to accelerate the production of the repetitive code that comprises 75-85% of the typical enterprise business application - the "glue" between the business logic programmer and the backing data servers and databases. It does so by applying a knowledge base of General Expansion Rule (GEL) files following specific XML document schemas and semantic constraints to a Business Application Model (BAM), which is an enhanced ERD constrained by a customized schema definition available here: mssbam-2.13.xsd, with 3.1 using an updated mcf/3.0.1/xsd/mssbam-3.1.xsd specification adapted to the new capabilities of the customized CFCore Engine eventualy implemented for the 3.1 BAM.

The BAM defines the tables, indexes, and relationships of an ERD, including Superclass, Container/Component, Parent/Child, and Lookup relationships. It also provides the means of encapsulating customizations of the output source code, with the approach changing radically compared to the old 2.0.12/2.0.13 approach of hard-coded verbs in the custom engine. Both 2.0.13 and 3.1 implement "tweaks" (see prior section on enhancements and improvements.)

I invite you to read through the prototype architecture and philosophies, and through the 2.11 documentation. Although seriously stale about the details, most of the key concepts and descriptions about how the CFCore engine works as described in the 2.11 documentation are still valid to this day.

I'm currently refreshing and refactoring my code base to support generic Java clients, take advantage of the latest Java features, provide back-end Spring web services initially using JSON to replace the old XML message protocol, and later replacing JSON with a byte-stream client if at all feasible. Along the way, the databases have been split up into isolated instances with the Obj layer providing a unified view of the component database hierarchy. That way each data silo can retain control over its own security concerns, subject to unified identification and authorization of users through the security schema (CFSec) packages and services at the bottom of the stack. Check the development log below for details about the progress on the 3.1 code base to date. Remember that while the 3.1 code base is not complete, you certainly can define a custom application model, produce fractal code for it using the same rules, and have a preview of the 3.1-compliant code for your model. Just remember to reference the CFSec security schema before anything else in your custom application model (BAM.)

The security schema itself may well become an OAuth2 service provider by the time I'm done, too, which I don't expect to be for quite some time, possibly not before 2027. As such, this code is very much in a state of flux.
